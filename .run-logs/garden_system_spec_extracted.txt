1. PROJECT OVERVIEW
This document is a strict technical specification for a web-based Personal Garden/Allotment Information System. Every requirement, entity, use case, and business rule stated here is derived directly from the original BBP thesis. Implementations MUST conform exactly to this specification.

1.1 Purpose
The system enables individual garden owners (or groups of owners) to plan, monitor, and manage personal garden or allotment activities. It replaces fragmented, manual processes (notes, separate apps, memory) with a unified, data-driven platform.

1.2 Key Capabilities
Interactive plot plan with visual zone & plant management
Multi-year planning history with crop rotation support
Automated recommendation-based action calendar
Plant condition tracking with status history
Integration with external weather forecast API (Meteo.lt)
Integration with external plant care API (Perenual)
Inventory management (tools & materials)
Multi-user collaboration on a single plot (with role-based access)
Community sharing of plot plans
Harvest registration and historical analysis
PDF export of plot plans

2. USER ROLES & ACTORS
The system has three user roles:

Role
Description
Guest (Svečias)
Unauthenticated visitor. Can only access the registration and login pages.
Garden Owner (Daržo savininkas)
The primary authenticated user. Can manage their own plots, plants, calendar, inventory, community features.
Administrator (Administratorius)
Has all garden owner capabilities plus the ability to view, search, and delete any user account.

NOTE: Garden Owner is a specialization (inheritance) of User. The system stores a role field on the User table ('owner' or 'admin').

3.1 User Account Management Sub-system
UC-01: Register (Registruotis)
Use Case
UC-01: Register
Goal
Create a new user account in the system.
Pre-condition
User is NOT authenticated in the system.
Trigger
Guest selects the 'Register' function.
Post-condition
A new user account is created and the user is redirected to the login page. OR an error message is shown if registration fails.
Main Flow
1. Guest initiates registration. 2. System displays registration form. 3. Guest fills in required data. 4. System validates form completeness and correctness. If invalid → show error, allow correction (go to step 3). 5. System checks email uniqueness. If already registered → show error, allow correction (go to step 3). 6. System saves user data, creates account, shows success message. 7. System redirects user to login page.

UC-02: Login (Prisijungti)
Use Case
UC-02: Login
Goal
Authenticate a registered user and grant access to the system.
Pre-condition
User has a registered account.
Trigger
User selects the 'Login' function.
Post-condition
User is authenticated and redirected to the main page. OR error message shown if login fails.
Main Flow
1. User initiates login. 2. System displays login form. 3. User may choose 'Forgot password' — if so, UC-04 is executed, then return here. 4. User enters credentials. 5. System validates credentials. If invalid → show error, allow retry. 6. System authenticates user, creates session, redirects to main page.

UC-03: Logout (Atsijungti)
Use Case
UC-03: Logout
Goal

Pre-condition
User is authenticated.
Trigger

Post-condition

Main Flow

UC-04: Forgot Password (Priminti slaptažodį)
Use Case
UC-04: Forgot Password
Goal
Allow a user to recover a forgotten password via email verification code.
Pre-condition
User has a registered account.
Trigger
User selects the 'Forgot Password' function.
Post-condition
Password is updated and user is shown the login form. OR error message shown.
Main Flow
1. User initiates forgot password. 2. System displays email input form. 3. User enters email. 4. System checks if email exists in DB. If not → error, allow retry. 5. System generates reset code, triggers email send via email server boundary. 6. Email server sends message to user. 7. User receives email with code. 8. System shows code input field. 9. User enters code. 10. System verifies code. If wrong → error, allow retry or regenerate code. 11. System shows new password form. 12. User enters new password. 13. System validates password meets requirements. If not → error, allow retry. 14. System updates password in DB, shows success, redirects to login.

UC-12: Mark Plant Rotation (Pažymėti augalų rotaciją)
Use Case
UC-12: Mark Plant Rotation
Goal
Register crop rotation decisions — assign new planting locations for plants in zones.
Pre-condition
Owner is authenticated and has at least one plot plan with defined plant zones and plants.
Trigger
Owner selects 'Mark Plant Rotation'.
Post-condition
Updated plant rotation info saved in system with new locations. Rotation data available for planning history analysis.
Main Flow
1. Owner initiates rotation marking. 2. System shows existing plot plans. 3. Owner selects plan. 4. System loads plan data, retrieves plant info, checks rotation recommendations (whether any plants were in the same zone in previous seasons). 5. If violations detected → system notifies owner of alternative planting locations. 6. Owner selects a plant and specifies new planting location/zone. 7. System checks zone suitability (against plant requirements) and compatibility with other plants in zone. If unsuitable → warning shown. 8. Repeat steps 6–7 for each plant as needed. 9. Owner confirms all rotation decisions. 10. System validates compatibility, saves updated rotation info, shows confirmation. OR if rejected: system reverts to previous state.

UC-16: Identify Plant Condition (Identifikuoti augalo būklę)
Use Case
UC-16: Identify Plant Condition (System-initiated)
Goal
Automatically determine and update a plant's current condition based on growth duration, biological parameters, and existing data.
Pre-condition
A plant exists in the system with a planting date and assigned biological parameters.
Trigger
System initiates 'Identify Plant Condition' automatically.
Post-condition
Plant is assigned an updated condition matching its development stage or health status. Saved in system.
Main Flow
1. System retrieves date of checking period and plant's planting date. 2. Calculates growth duration. 3. System retrieves plant's biological parameters from Plant_care dataset. 4. Checks if plant has disease flag set. If YES → set condition = 'diseased', save, end. 5. If NO disease: system evaluates development stage thresholds: — If germination start period not yet passed → 'planted'; — If germination ongoing → 'germinating'; — If growing period → 'growing'; — If flowering period → 'flowering'; — If maturity period → 'mature'. 6. Additionally: if plant is annual AND maturity end passed → condition = 'dried'. If plant is perennial AND dormancy period → condition = 'regenerating'. 7. System saves updated condition.

Plant condition states (enumeration): planted, germinating, growing, flowering, mature, dried, diseased, regenerating

UC-20: Analyze Garden Results (Analizuoti daržo rezultatus)
Use Case
UC-20: Analyze Garden Results
Goal
Evaluate garden activity results from multiple analytical perspectives.
Pre-condition
Owner is authenticated, has at least one plot plan, and has accumulated historical garden data.
Trigger
Owner selects 'Analyze Garden Results'.
Post-condition
Analysis report generated and saved. Presented to user with selected analysis results and overall plot utility indicators.
Main Flow
1. Owner initiates analysis. 2. System shows available plot plans. 3. Owner selects plan. 4. System requests analysis type selection. Owner chooses: (A) Planning decision analysis → system retrieves historical planning data, extracts plant selections per zone per season, checks rotation violations, calculates plan change frequency. (B) Plant condition analysis → system retrieves historical condition records, sorts chronologically, identifies condition changes, finds critical deterioration points, evaluates improvement trends after maintenance actions. (C) Harvest analysis → system retrieves historical harvest records, calculates total harvest quantities, identifies trends, identifies best-yielding plants, compares actual vs planned harvest. 5. After selected analysis branch: system calculates overall plot utility indicators, generates final summary report, presents to owner.

8. SYSTEM ARCHITECTURE
The system uses a three-layer MVC architecture:

8.1 Layer Structure
UI Layer (Frontend — React): All application windows/pages organized by functional area. Handles user interaction, displays data, sends requests to controller layer.
Controller Layer (Backend — Laravel): Handles all business logic, request processing, validation, coordination between UI and data layers. Organized by the same functional sub-systems.
Data Layer (PostgreSQL via Laravel Eloquent ORM): All domain entities and their relationships as defined in Section 5.

8.4 Key Business Rules
A user can own zero or more plots. A plot can have multiple owners (N:M via access_rights).
A plot has one or more plant zones. Each zone belongs to exactly one plot (1:N).
A plant zone can contain zero or more plants. Each plant belongs to one zone.
One plot can have multiple task calendars (one per planning period).
A task calendar has zero or more tasks.
A plant can have zero or more condition history records.
A plant zone has rotation history records tracking which plants were grown when.
Each garden owner has their own inventory (not shared between owners).
Community posts can reference a plot. A plot can be marked as shareable (share = true).
Password reset: code generated, stored in reset_code, sent by email, verified, then used once and deleted.
Plant condition is both manually recordable (UC-17) AND automatically computed by the system (UC-16).
Disease overrides developmental stage: if disease=true → condition='diseased' regardless of growth stage.
Annual plants (reusable=false): when growth ends → condition='dried'.
Perennial plants (reusable=true): when dormancy period → condition='regenerating'.
Calendar generation requires: plot data + plant conditions + weather forecast. Generates task list per date.
Rotation check: system warns if a plant was in the same zone in a previous season (uses rotation_history).
PDF export collects plot + zone + plant data and generates downloadable document.

11. RECOMMENDATION CALENDAR GENERATION ALGORITHM
The calendar generation (UC-14) uses the following logic:

INPUTS: Plot ID, planning period (start_date → end_date), plant list with conditions, weather forecast data, plant_care data, inventory items

FOR EACH DATE in planning period:
Fetch weather forecast for date. If weather is extreme (see thresholds) → generate weather-warning tasks.
FOR EACH PLANT in plot:
Call Identify Plant Condition (UC-16) → determine current condition_type.
Retrieve plant_care record for this plant type.
Check if any tasks are needed: evaluate watering_interval_days, fertilizing_interval_days, pest_check_interval_days.
Check weather conditions: if precipitation > rain_skip_threshold_mm → skip watering. If temperature < frost_temp_threshold_c → add frost protection task. If temperature > heat_extra_water_temp_c → add extra watering task. If wind > wind_protection_kmh → add wind protection task.
Generate Task entries with appropriate task_type, name, related plant/zone, and item requirements.
Check inventory: for each generated task needing materials, call check_if_enough() on Inventory_item. If insufficient → flag the task with needed materials info.

SAVE: Create task_calendar record, save all Task records linked to it. Associate weather_forecast records.
12. IMPLEMENTATION NOTES & CONSTRAINTS
Backend must use Laravel's Eloquent ORM for all database operations. No raw SQL unless absolutely necessary for complex analytics queries.
All relationships must be enforced via Eloquent relationships AND PostgreSQL foreign key constraints.
Use Laravel migrations for all schema changes. No manual DB alterations.
Use Laravel's built-in authentication (Sanctum or session-based) for user auth.
All forms must use Laravel server-side validation. Frontend validation is additional, not a replacement.
React frontend communicates with Laravel backend via REST API endpoints.
The plot geometry field uses JSON to store map boundary coordinates.
PDF export must include: plot name, all zones with names, all plants per zone with names and conditions.
Community: only plots with share=true are visible in community browsing.
Access control: check role in access_rights for every plot operation. Editor can modify. Viewer can only read.
Admin role is set in users.role field. Separate from plot-level access_rights.
Rotation check must query rotation_history for the same plant_zone_id and overlapping time periods.
Planning history: each time a plot is significantly modified, save a version snapshot for history.
Plant condition is auto-identified server-side — no user action needed for UC-16. It is triggered from calendar generation and task processing.
Weather data: fallback to most recent weather_forecast record if API call fails.
All monetary/quantity fields use DOUBLE PRECISION in PostgreSQL.
System must be accessible on both desktop browsers and mobile browsers (responsive CSS).
Page load time target: ≤ 3 seconds for all primary pages.
